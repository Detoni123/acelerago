### Arquitetura

- Cada imagem de slide vai pra `${userId}/${carouselId}/slide-${idx}.jpg`
- Logo da médica vai pra `${userId}/_logos/logo.png`
- DB armazena só `imageUrl` (URL pública do Supabase Storage), nunca mais base64
- `imageVersion` (timestamp) usado como cache-buster estável: `?v=${imageVersion}`

### Helpers adicionados (gerador-carrossel.html ~linhas 125-180)

| Função | Propósito |
|--------|-----------|
| `slideImgSrc(slide)` | Retorna `data:image/png;base64,...` ou `URL?v=version` |
| `hasSlideImage(slide)` | True se slide tem `imageData` ou `imageUrl` |
| `b64ToCompressedBlob(b64)` | Canvas-comprime PNG → JPEG quality 0.85 (~70% menor) |
| `uploadSlideImage(b64, userId, carouselId, idx)` | Upload comprimido → retorna `{url, version}` |
| `uploadClientLogo(b64, ownerId)` | Upload logo da médica |
| `listCarouselFiles(userId, carouselId)` | Lista arquivos de pasta no Storage |
| `removeStorageFiles(paths)` | Remove arquivos (cleanup) |

### Fluxo de save (handleSaveCarousel)

1. Mostra "Subindo imagem X de Y..." na UI
2. Para cada slide com `imageData`, upload pro Storage com `upsert: true`
3. Persiste `imageUrl` + `imageVersion` no DB; `imageData` é stripado
4. Cleanup: lista arquivos no Storage que não correspondem a slides atuais e remove
5. Update da row em `saved_carousels` (próprio user via RLS, outro via adminCall)
6. Sincroniza state em memória (slidesForState mantém base64 pra preview imediato)

### Fluxo de load (handleLoadCarousel)

1. Prefere `imageUrl` quando presente
2. Fallback pra localStorage se for carrossel legado (sem `imageUrl`)
3. Se houver legacy: chama `migrateLegacyImages` em background (upload silencioso → update DB → toast de confirmação)

### Fluxo de delete (handleDeleteCarousel)

1. Lista arquivos do Storage da pasta do carrossel
2. Remove esses arquivos
3. Deleta a row do DB
4. Remove cache localStorage

## Bloco 2: Sistema de logo da médica

### Schema

- Coluna `doctor_profiles.logo_url` armazena URL do logo no Storage
- Mapeado em `profileFromDB` / `profileToDB` como `logoUrl`

### Upload no perfil

- Seção no `ClientModal` (após "Foto da Médica")
- Aceita imagem, mostra preview com bordinha laranja
- Sobe imediatamente pro Storage (não espera salvar perfil)
- PNG transparente funciona melhor

### Campos do slide para logo

| Campo | Range | Default | Descrição |
|-------|-------|---------|-----------|
| `logoEnabled` | bool | false | Mostrar logo neste slide |
| `logoX` | 0–100 | 50 | Posição horizontal (% do canvas, centro do logo) |
| `logoY` | 0–100 | 90 | Posição vertical (% do canvas, centro do logo) |
| `logoSize` | 5–80 | 25 | Largura (% do canvas) |
| `logoOpacity` | 0–100 | 100 | Transparência |
| `logoPosition` | string | — | DEPRECATED (auto-migrado pra X/Y) |

### Renderização

- **JSX**: `<img>` absolutamente posicionado dentro do slide com `transform: translate(-50%, -50%)` (anchor no centro)
- **Canvas**: `computeLogoPos(slide, SW, SH, naturalW, naturalH)` calcula box, `ctx.drawImage` por cima de tudo
- Pré-carrega logo no início do `renderToCanvas` pra ter `naturalWidth/naturalHeight`

### Botão "Aplicar este logo a todos os slides"

- Copia `{logoEnabled, logoX, logoY, logoSize, logoOpacity}` pra todos os slides via `onApplyLogoAll`

### Admin editando outras médicas

- Botão "✏ Perfil" abre o perfil do `client` ativo (não sempre o admin)

## Bloco 3: Controle de offsets independentes por elemento

Substituiu o `textOffsetY` (que movia todos os elementos junto) por offsets individuais:

| Campo | Range | Afeta |
|-------|-------|-------|
| `titleOffsetY` | -400 a +400 | Apenas título |
| `imageFrameOffsetY` | -400 a +400 | Apenas frame da imagem (não confundir com `imageOffsetY` que move dentro do frame) |
| `bodyOffsetY` | -400 a +400 | Apenas texto |
| `textOffsetY` (legado) | -400 a +400 | Move o grupo todo (backward compat, oculto se não tem valor) |

### Implementação no canvas

Usa `ctx.save() + ctx.translate(0, offset) + draw + ctx.restore()` envelopado em helper `withShift`. O offset afeta visualmente sem alterar o fluxo do layout (ty da próxima posição).

### Implementação no preview

Cada elemento individual recebe `transform: translateY(${offset}px)` no inline style. O `textOffsetY` do grupo continua aplicado no flex parent.

## Bloco 4: Match Canvas ↔ Preview (a parte mais complexa)

A canvas renderiza diferente do preview. Vários bugs combinados causavam diferenças visíveis: texto sobrepondo logo, fonte serif aparecendo no export, "g" e "u" sozinhos em linhas separadas, layout deslocado.

### Bug 1: Quebras de linha diferentes entre canvas e CSS

**Diagnóstico**: `ctx.measureText` mede texto sem kerning, CSS aplica kerning natural. Resultado: canvas quebra texto em mais linhas que preview.

**Fix**: criou `measureCSSLines(text, maxW, fontWeight, fontSize, fontFamily, letterSpacing)` que cria `<div>` invisível idêntico ao preview, usa `range.getClientRects()` pra descobrir onde o navegador quebra cada linha.

**Bugs encontrados no caminho:**

- `word-wrap: break-word` no div quebrava palavras no meio ("gestação" → "g" + "estação"). Fix: `overflow-wrap: normal; word-break: normal; word-wrap: normal`.
- `getClientRects()` retorna **2 rects por linha visual** (um pro texto, outro pro ponto de wrap). Fix: deduplicar por `top` (tolerância 2px) antes de iterar.
- `letter-spacing: 0px` desabilitava kerning. Fix: só seta `letterSpacing` quando o slide tem valor explícito (caso contrário deixa default `normal`).

### Bug 2: Fontes diferentes entre canvas e preview

**Diagnóstico**: `ctx.font` shorthand falha quando o nome da fonte tem espaço (ex: "Playfair Display"). Browser cai pra fonte serif padrão. Preview ignorava `client.fonte` e hardcodava Sora.

**Fix**:
- Variável `font` no canvas agora é a chain CSS completa: `"Nome", "Sora", sans-serif`
- `measureCSSLines` reusa a chain (não duplica aspas se já tem vírgula)
- Preview (`innerStyle.fontFamily`) usa `client.fonte` com fallback

### Bug 3: Layout estático no canvas vs flex dinâmico no preview

**Diagnóstico**: canvas usava `startY` fixo (140 top, 400 center, 700 bottom). Preview usa `justifyContent` que centraliza dinamicamente baseado no tamanho real do conteúdo. Para vAlign='center' com conteúdo curto, canvas posicionava ~250px abaixo do esperado.

**Fix (capa)** — `renderToCanvas` linhas ~680-770:
1. `measureCSSLines` pra headline e subtitulo
2. Calcula `contentH = destaqueH + headlineH + subtituloH + gaps`
3. `startY = SH - padBot - contentH - textOffsetY` (para `textPos='bottom'`)
4. Renderiza com `textBaseline: 'top'`
5. Bug colateral resolvido: "1 / deslize →" estava combinado no topo. Agora "1 / X" no topo direito, "deslize →" na barra de rodapé.

**Fix (conteúdo)** — `renderToCanvas` linhas ~870-1030:
1. Mede `titleLines` e `bodyLines`
2. `contentH = items.filter(>0).sum() + gaps`
3. `startY` calculado conforme vAlign: top=`padTop`, center=`padTop + (innerH-contentH)/2`, bottom=`padTop + (innerH-contentH)`
4. `textBaseline: 'top'` consistente
5. Gaps explícitos de 36px (antes era 30+80 fora do padrão)

### Bug 4: Tamanhos de fonte diferentes

**Diagnóstico**: `adaptFs(slide.titulo, 76, 52)` no canvas reduzia título pra 52px em textos longos. Preview sempre usava 72px. Layout completamente diferente.

**Fix**: removidas as chamadas a `adaptFs`. Canvas agora usa os mesmos defaults do preview:
- Capa headline: 78
- Conteúdo título inline/texto: 72; lateral: 60
- Conteúdo body inline: 44; lateral: 40; texto: 58
- Conteúdo destaque (capa): 46 (preview) — antes era 52

### Bug 5: Line-heights diferentes

| Elemento | Canvas (antes) | Preview | Canvas (agora) |
|----------|---------------|---------|----------------|
| Title inline | 1.20 | 1.18 | 1.18 |
| Body inline | 1.55 | 1.65 | 1.65 |
| Body texto | 1.52 | 1.5 | 1.5 |
| Body lateral | 1.55 | 1.65 | 1.65 |
| Gap entre itens | 30 + 80 | 36 | 36 |
